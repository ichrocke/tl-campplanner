 0. MANAGEMENT-SUMMARY – WAS ZUERST?
================================================================================

Die drei drängendsten Themen (in dieser Reihenfolge):

 1) STORED XSS in der Collab-Version (KRITISCH, Sicherheit).
    Objekt-/Gruppen-/Layer-/Template-Namen anderer Raum-Teilnehmer werden
    ungefiltert per innerHTML gerendert. Ein Teilnehmer kann Schadcode im
    Browser aller anderen ausführen. Fix ist mechanisch klein (esc()-Helfer
    existiert bereits, muss nur überall angewandt werden). -> Teil B, X1.

 2) DATENVERLUST in der Collab-Version (KRITISCH, Funktion).
    Genau das vom Nutzer gemeldete Symptom ("erscheint auf beiden Zeltplätzen,
    vorher Gezeichnetes verschwindet") ist durch mehrere sich verstärkende
    Sync-Bugs am Code reproduzierbar (geteilter Timer, Index-statt-ID-Ops,
    Versionsfortschreibung ohne State-Übernahme). -> Teil C, C1–C4.

 3) FEHLENDE ZUGRIFFSKONTROLLE im Backend (HOCH, Sicherheit).
    Raum-ID ist das einzige Token; schwach generiert; CORS offen;
    ADMIN_KEY leakt über URL/Referer an Karten-CDNs. -> Teil B, S1–S4.

Für den Standalone-Betrieb (ohne Raum) ist das Tool grundsätzlich solide;
die größten Risiken dort sind Autosave-Datenverlust bei großen Bildern und
"klebende" Drags. -> Teil D.

 TEIL C – COLLAB / RAUM: DATENVERLUST & KONSISTENZ
================================================================================
 (Erklärt die Nutzer-Meldung im Raum-Betrieb. Alle am Code verifiziert, wo [V].)

ARCHITEKTUR-KONTEXT
  Hybrid-Protokoll: Objekt-Ops (add/update/remove, adressiert per siteIdx +
  objectId) + Full-State-Push (Last-Writer-Wins mit Versionsprüfung) als
  Fallback. Genau an den Nahtstellen dieser zwei Pfade liegen die Bugs.

(ERLEDIGT & LIVE-VERIFIZIERT 2026-07-09: C1 (room-ops gibt gemergten State
 zurück, Client übernimmt ihn), C2/C3 (serialisierter runSync: Ops flushen +
 danach Full-Push; Strukturänderungen gehen nicht mehr verloren), C4 (Ops mit
 siteId; Server-Lookup per ID, Index-Fallback; live getestet: Objekt landete
 per siteId auf der richtigen Site), C5(a) (Remote-Importe nicht mehr im
 Undo-Stack). Client-Logik per Node-Harness geprüft, Server per Live-API-Test.
 Offen bleibt C5(b): op-basiertes Undo, damit eigenes Ctrl+Z fremde Arbeit
 nicht global überschreibt.)

C2  KRITISCH  Geteilter _opsTimer: pushOp cancelt den Full-Push
    [V collab.js:339 vs. 347-348]  pushState() plant doFullPush auf _opsTimer
    (150ms); jede Op innerhalb dieser 150ms macht clearTimeout(_opsTimer) und
    plant flushOps -> der Full-Push findet NIE statt. Strukturänderungen
    (Tab löschen/anlegen/duplizieren, Append-Import, Undo, Drag-Commit) gehen
    verloren. Das ist der Kern von "erscheint auf beiden Zeltplätzen":
    Tab-Löschung erreicht den Server nie, folgende Op landet per Index auf dem
    (serverseitig noch existierenden) falschen Platz.
    FIX: zwei getrennte Timer (_fullPushTimer, _opsTimer) + _needFullPush-Flag,
    das nach erfolgreichem flushOps einen ausstehenden Full-Push nachholt.

C3  KRITISCH  pushState mit Pending-Ops sendet NUR die Ops
    [V collab.js:334-336]  Löst eine Full-State-würdige Änderung (Rename, Layer/
    Template, Drag-Commit, Undo) pushState aus, während Ops in der Queue liegen,
    wird nur geflusht – die Änderung selbst nie übertragen (stille Divergenz).
    FIX: wie C2 (_needFullPush) oder Strukturänderungen als eigene Ops.

C4  KRITISCH  Ops adressieren Sites per INDEX statt ID
    [V state.js addObject pushOp {siteIdx:_activeSiteIndex}]  Server:
    room-ops.php verwirft ungültigen Index STILL (continue -> Datenverlust ohne
    Fehler). Bei Tab-Löschen/duplicateSite verschieben sich Indizes -> Ops
    landen auf falscher/fehlender Site; gleiche Objekt-ID kann auf zwei Sites
    landen ("auf beiden Zeltplätzen"). migrateSite setzt aktuell KEINE s.id für
    Alt-States [V].
    FIX: Ops zusätzlich mit siteId senden; Server-Lookup per id, Fallback auf
    siteIdx für Altclients; nicht gefundene Site -> Full-Refetch statt still
    verwerfen; migrateSite um "if(!s.id) s.id=generateId()" ergänzen.

C5  HOCH  Undo/Redo verwirft im Raum global fremde Arbeit
    [V state.js undo(): _listeners.forEach(fn=>fn(false)) -> Full-Push]
    Zudem landen Remote-Updates im Undo-Stack (importJSON ruft notify(false,..)
    = skipUndo=false). Ctrl+Z stellt einen Snapshot vor dem letzten
    Fremd-Update her und pusht ihn -> löscht die Objekte des anderen.
    FIX (a, Zweizeiler): Remote-Imports mit skipUndo=true (bei skipSync===true
    notify(true, skipSync)). FIX (b, sauber): op-basiertes Undo nur eigener Ops.

C6  HOCH (TEILWEISE 2026-07-09)  Full-Push-Konflikt verwirft lokale Änderungen
    Serialisierter Sync mildert das (Ops werden separat behbuilt + gemergter
    State übernommen), aber ein rein struktureller lokaler Full-Push, der mit
    einer fremden Änderung KOLLIDIERT, kann weiterhin verloren gehen.
    REST-FIX: verlorene strukturelle Änderung nach Konflikt-Merge erneut
    anwenden + erneut pushen (bzw. Strukturänderungen als Ops, s. C16).

(ERLEDIGT 2026-07-09: C7 (Ops bleiben bei Fehler in der Queue + Retry –
 Node-Harness bestätigt), C8 (disconnect() setzt allen Sync-State zurück),
 C9 (lokale Änderungen während Raum-Sperre werden beim Entsperren nachgesendet).)

(C10 ERLEDIGT 2026-07-09: room-sse.php + room-cursors.php nutzen jetzt
 INSERT ... ON DUPLICATE KEY UPDATE statt REPLACE.)

C11 MITTEL  Cursor/Presence ohne Site-Bezug
